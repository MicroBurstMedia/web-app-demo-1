The Next.js 16 Network Boundary

Architecture context

Next.js 16 moves away from the implicit middleware.ts proxy pattern toward an explicit proxy.ts network boundary. Outbound API traffic is no longer a side-effect of the middleware pipeline — it is a declared, testable module. The shift also forces a deliberate split between auth state (server-only) and UI state (client-safe).

Legacy pattern — middleware.ts (pre-16, implicit)

// middleware.ts — implicit proxy, deprecated in Next.js 16
export function middleware(req: NextRequest) {
  const token = req.cookies.get('session_token');
  // Silent URL rewrite — opaque, no explicit auth contract
  return NextResponse.rewrite(
    new URL(`/api${req.nextUrl.pathname}`, req.url)
  );
}
// ❌ Any route could reach the backend implicitly
// ❌ Token handling scattered across multiple files
// ❌ No testable boundary for auth headers

Failure hypothesis

HIGH RISKSAFARI / ITP

Safari's Intelligent Tracking Prevention (ITP) aggressively partitions localStorage in cross-origin iframes and private-browsing contexts. Storing a JWT session token in localStorage creates two distinct failure modes:

1 · Silent purge. ITP clears the partition without warning, logging the user out mid-session with no network error — the token simply vanishes. 2 · Tab isolation in private mode. A deep-link opens a new unauthenticated tab; the previous session is invisible to it. Both failures are unreproducible on Chrome/Firefox, making them nearly impossible to diagnose from server logs.

Secondary risk: any XSS vector on the page can exfiltrate a localStorage token in a single line. An httpOnly cookie is completely opaque to JavaScript — the attack surface does not exist.

proxy.ts — explicit boundary + cookie split

// src/proxy.ts — Next.js 16 explicit network boundary
import { cookies } from 'next/headers';

export async function proxyRequest(
  targetPath: string,
  init?: RequestInit
) {
  // httpOnly cookie — never readable by client JS (XSS-proof)
  const token = (await cookies()).get('session_token')?.value;

  return fetch(`${process.env.API_BASE_URL}${targetPath}`, {
    ...init,
    headers: {
      'Authorization': `Bearer ${token}`,
      'Content-Type': 'application/json',
    },
  });
}

// Server Action: set session cookie on login (server-only)
export async function setSessionCookie(token: string) {
  (await cookies()).set('session_token', token, {
    httpOnly: true,   // JS-opaque, ITP-safe
    secure: true,
    sameSite: 'strict',
    maxAge: 60 * 60 * 24,
  });
}

// localStorage: non-sensitive UI state only (Safari-safe)
localStorage.setItem('ui_theme', 'dark');
localStorage.setItem('pinned_league', leagueId); // no auth value

The API "Cascading 429" Failure Hypothesis

Environment constraint

API-Football enforces a hard cap of 500 requests/minute. During peak match windows — Champions League knockouts, tournament group stages — concurrent live-score polling from multiple browser tabs can exhaust this budget within seconds. A single unchecked polling loop is enough to cascade into platform-wide 429s.

Current unguarded route

// src/app/api/football/live/route.ts — no circuit breaker
export async function GET() {
  const res = await fetch(FOOTBALL_API_URL, {
    headers: { 'X-RapidAPI-Key': process.env.API_KEY! }
  });
  return NextResponse.json(await res.json());
}
// ❌ No X-RateLimit header inspection
// ❌ No circuit breaker — crashes at 429 with no warning
// ❌ TanStack Query default retry fires 3x on failure → tripling load

Cascade scenario — tournament day

CRITICAL

08:00 UTC · 12 group stage matches kick off simultaneously. TanStack Query refetch intervals across 4 open tabs poll at 30 s each. At 3 concurrent users that is ~48 req/min baseline. Add Vercel edge revalidation and the effective rate climbs to ~300 req/min before 09:00.

09:30 UTC · A flurry of goals triggers manual refreshes. The counter crosses 500. API returns 429. TanStack Query's default retry: 3 fires per hook — tripling the in-flight request count. Every user sees stale scores simultaneously. The cascade is complete and self-reinforcing.

Retry storms amplify 429s into sustained blackouts. The root cause is the absence of a pre-emptive Cool Down signal — the system should brake at 50 remaining requests, not after hitting zero.

Circuit breaker — X-RateLimit header interception

// src/app/api/football/live/route.ts — with circuit breaker
import { NextResponse } from 'next/server';

const COOLDOWN_THRESHOLD = 50; // brake before the wall

export async function GET() {
  const res = await fetch(FOOTBALL_API_URL, {
    headers: { 'X-RapidAPI-Key': process.env.API_KEY! }
  });

  // Inspect upstream rate-limit headers on every response
  const remaining = parseInt(
    res.headers.get('X-RateLimit-Requests-Remaining') ?? '500'
  );
  const resetAt = res.headers.get('X-RateLimit-Requests-Reset');

  if (remaining < COOLDOWN_THRESHOLD) {
    // Signal Cool Down before the 429 — serve last-good data
    return NextResponse.json(
      { cooldown: true, resetAt, data: await res.json() },
      { status: 200, headers: { 'X-Cooldown': 'true' } }
    );
  }
  return NextResponse.json(await res.json());
}

// TanStack Query — respect cooldown flag, zero retries on 429
const { data } = useQuery({
  queryKey: ['live-scores'],
  queryFn: fetchLiveScores,
  refetchInterval: (q) => q.state.data?.cooldown ? false : 30_000,
  retry: 0,  // never amplify a 429
});
SLIDE 03 / 05

Backend Integrity & The Service Layer

Match update data flow

API-Football webhooks deliver match events in UTC. The FastAPI router receives the payload, persists it via asyncpg, and the frontend fetches and renders it. The critical architectural question is: at which layer does kickoff_utc → Asia/Taipei conversion occur?

Anti-pattern — timezone conversion in the router

# routers/matches.py — tz logic leaks into the transport layer
@router.get("/matches/{match_id}")
async def get_match(match_id: int, db: AsyncSession = Depends(get_db)):
    result = await db.execute(select(Match).where(Match.id == match_id))
    match = result.scalar_one_or_none()
    # ❌ Conversion duplicated in every router that touches a match
    taipei = match.kickoff_utc.astimezone(ZoneInfo("Asia/Taipei"))
    return {"kickoff": taipei.isoformat()}

Logic leakage — data drift hypothesis

DATA DRIFT

When timezone conversion lives in the router, every new route that surfaces match data must independently re-implement the offset. Within weeks three variants emerge: one uses ZoneInfo("Asia/Taipei") correctly; a second uses a hardcoded timedelta(hours=8) which breaks on DST edge cases; a third passes raw UTC to the frontend, which converts using the browser's locale — producing three different displayed times for the same kickoff across scoreboard, notifications, and calendar export.

This drift is near-impossible to detect in unit tests because each implementation is locally correct. It surfaces only when a user receives a push notification 30 minutes before a match that has already started.

Service Layer pattern + async SQLAlchemy via asyncpg

# services/match_service.py — single source of truth for tz conversion
from zoneinfo import ZoneInfo
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select

TAIPEI = ZoneInfo("Asia/Taipei")  # defined once, referenced everywhere

class MatchService:
    async def get_match(
        self, db: AsyncSession, match_id: int
    ) -> MatchSchema | None:
        # Async SQLAlchemy — asyncpg handles the connection pool
        result = await db.execute(
            select(Match).where(Match.id == match_id)
        )
        match = result.scalar_one_or_none()
        if not match:
            return None

        # UTC → Taipei: happens ONLY here. Never in router, never client-side
        taipei_time = match.kickoff_utc.astimezone(TAIPEI)

        return MatchSchema(
            id=match.id,
            home_team=match.home_team,
            away_team=match.away_team,
            kickoff_taipei=taipei_time,    # localised once
            kickoff_utc=match.kickoff_utc, # raw UTC preserved for logs
        )

# routers/matches.py — now thin; zero tz logic
@router.get("/matches/{match_id}")
async def get_match(match_id: int, svc: MatchService = Depends()):
    return await svc.get_match(db, match_id)

Caching Strategy & cacheLife

Next.js 16 cacheLife API

cacheLife() replaces raw revalidate integers with named semantic profiles declared in next.config.ts. Paired with the 'use cache' directive, caching intent lives at the function level — not buried in individual fetch() options. This makes cache policy auditable, consistently applied, and easy to tune as rate-limit budgets change.

Two profiles are critical for a live football platform: a seconds profile for live scores (aggressive refresh, high cost per request) and an hours profile for league standings (static data, near-zero ongoing cost).

Budget burn — uniform TTL failure

BUDGET BURN

Applying revalidate: 60 uniformly across all football routes produces simultaneous failures in opposite directions: live match scores become 60 s stale during stoppage time (unacceptable), while league standings — which change at most twice per match day — are re-fetched every minute, burning ~1,440 req/day on data that could be cached for an hour.

During a 16-team tournament, uniform 60 s polling consumes roughly 80% of the 500 req/min daily budget on static standing data before a single live event is tracked. The circuit breaker from Slide 2 fires constantly — not because of real load, but because of cache misconfiguration.

Tiered cacheLife — live vs static budget split

// next.config.ts — declare named profiles
const config: NextConfig = {
  experimental: {
    cacheLife: {
      'live-event': { stale: 0,   revalidate: 15,   expire: 30   },
      'standings': { stale: 300, revalidate: 3600, expire: 86400 },
      'fixtures':  { stale: 60,  revalidate: 900,  expire: 7200  },
    }
  }
};

// app/live/scores.tsx — "seconds" profile, high-priority budget
async function getLiveScore(matchId: string) {
  'use cache'
  cacheLife('live-event')  // revalidates every 15 s, stale:0 = always fresh
  const res = await proxyRequest(`/live/${matchId}`);
  return res.json();
}

// app/standings/[leagueId]/page.tsx — "hours" profile, near-zero cost
async function getStandings(leagueId: string) {
  'use cache'
  cacheLife('standings')  // revalidates hourly → ~24 req/day per league
  const res = await proxyRequest(`/standings/${leagueId}`);
  return res.json();
}
// Budget arithmetic: 10 live matches × 4 req/min = 40 req/min live cost
// 20 leagues × 1 req/hr = ~0.3 req/min standing cost → 460 req/min headroom

The UV & Dependency Management Audit

Environment baseline

uv replaces pip, pip-tools, and virtualenv in a single Rust-native binary. It reads pyproject.toml as the single source of truth and produces a deterministic uv.lock that is platform-portable and sub-second to install. uv sync runs in CI and the Docker build stage, guaranteeing identical dependency trees across dev, staging, and production.

# pyproject.toml — canonical dependency declaration
[project]
name = "football-api"
version = "0.1.0"
requires-python = ">=3.12"

dependencies = [
    "fastapi>=0.115.0",
    "sqlalchemy[asyncio]>=2.0.36",
    "asyncpg>=0.29.0",
    "pydantic>=2.9.2",
    "uvicorn[standard]>=0.32.0",
    "alembic>=1.14.0",
    "python-jose[cryptography]>=3.3.0",
]

Silent drift hypothesis

SILENT FAILURE

Without a lock file, a pip install -r requirements.txt on a fresh 03:00 UTC deploy can silently pull a minor version bump that breaks an internal API. Common scenario: sqlalchemy 2.0.36 → 2.1.0 changes async session handling; asyncpg 0.30.0 alters connection pool defaults. The app boots, the health check passes, and a subtle ORM regression ships to production — with no diff to review.

With uv, the lock file captures the exact resolved graph including all transitive deps. Any change is an explicit, reviewable diff committed to source control — not a silent runtime surprise discovered by a user at kickoff.
